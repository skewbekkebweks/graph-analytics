# Аналитика над графом - LeaderRank

## 1. Метрика

**Как работает.** LeaderRank - не требующий выбора параметров вариант PageRank для ранжирования вершин ориентированного графа. К `N` вершинам добавляется ground-узел, двусторонне связанный со всеми. По расширенному графу степенной итерацией ищется стационарное распределение случайного блуждания, затем ранг ground-узла раздаётся поровну. Применяется для поиска влиятельных вершин в социальных сетях.

**Почему выбран.** LeaderRank был выбран, так как не требует подбора параметров и применяется в социальных сетях, рекомендательных системах

## 2. Варианты представления и обработки

LeaderRank - степенная итерация: за итерацию рёбра читаются один раз, в памяти нужен вектор рангов. Рёбра стримятся с диска.

1. **Потоковая обработка рёбер.** Вектор рангов `O(|V|)` в RAM, рёбра в бинарном виде читаются с диска. Просто, минимум I/O (`O(|E|)` за итерацию). Требует, чтобы вектор влезал в память.
2. **Блочное партиционирование.** Вершины делятся на `B` блоков, рёбра - в сетку `BxB`. В памяти держится один блок вектора `O(|V|/B)`. Работает, когда и вектор `O(|V|)` не влезает.

## 3. Выбранный метод и ограничения

Реализован **Вариант 1**. "Не влезает в память" относится прежде всего к рёбрам, а `O(|V|)` в память влезает на реальных примерах.

**Устройство.**
- Препроцессинг (2 прохода по CSV): отражение id в `0..N-1`, подсчёт исходящей и входящей степени, раскладка рёбер в бинарные шарды по вершине, к которой направлено ребро. Границы шардов выбираются по накопленной входящей степени: у шардов ~равное число рёбер (нет тяжёлых шардов от гипер-узлов).
- Итерация: параллельно по шардам. Все рёбра, направленные в конкретную вершину, лежат ровно в одном шарде. Значит, запись в `next[]` идёт без гонок.
- Корректность проверяется независимой реализацией на NumPy (`scripts/validate.py`). Совпадение на примерах `~5e-14`.

**Ограничения.**
- Вектор рангов `O(|V|)` обязан помещаться в RAM. При экстремальном `|V|` нужен Вариант 2.
- Масштабирование ограничено пропускной способностью памяти

## 4. Результаты

Машина: MacBook (Apple Silicon), 8 ядер, сборка `-O3`, `tol = 1e-6`. Время / пиковая память (RSS).

| Граф | Вершин | Рёбер | preprocess | run (8 потоков) | numpy |
|---|---|---|---|---|---|
| web-Google | 875 713 | 5.1M | 1.0 с / 46 МБ | 1.1 с / 35 МБ | 8.3 с |
| soc-LiveJournal1 | 4 847 571 | 69M | 11.6 с / 203 МБ | 49.5 с / 166 МБ | 503 с |

`numpy` - независимая реализация того же LeaderRank на NumPy (`scripts/validate.py`, тот же `tol`). Она служит и эталоном корректности, и точкой сравнения по скорости.

**Масштабируемость по потокам** (soc-LiveJournal1, фиксированные 100 итераций):

| Потоков | 1 | 2 | 4 | 8 |
|---|---|---|---|---|
| Время | 16.8 с | 10.5 с | 7.8 с | 8.7 с |
| Ускорение | 1.0x | 1.6x | 2.2x | 1.9x |

Ускорение выходит на плато 2.2x на 4 потоках: итерация memory-bound, поэтому дальнейшее увеличение числа потоков выигрыша не даёт.
